fdemmer commented on Dec 10, 2015

the issue with fewer results sounds like some thing caused by the threshold. i wrote that code a long time ago, but i think eg. when 'oldies' and '60s' is in the list, it removes 'oldies' because '60s' is way more popular than 'oldies'.

every tag from last.fm comes with a score of popularity.

those scores a weighed (multiplied by a factor to make some more important than others, eg. artist tags are more important than track tags). this is done for more consistency over the years, different albums by the same artist. think, a single very "poppy" track should not influence a classic rock band's tagging too much.

the idea of the threshold is to get the most "important" or "meaningful" tags. this is especially important with obscure artists or tracks with very little tags. then the scores vary a lot. i am not sure about how exactly the scoring at last.fm works, but it has to relate to how many people have given the tag. eg '60s' has a score of 80, because 8 people used the tag and 'oldies' just 10, because just 1 person did.

so by eliminating the tags with a very low score (compared to the top scoring tag! 80 vs 10), the plugin filters the "underperforming" tags, those that are deviations from the opinion of the masses, assuming those were probably wrong or user mistakes.

in your case both 'oldies' and '60s' are valid and probably accurate tags, but i have observed that with little tagged tracks or artists i get "noise" in my tags, even contradicting tags.

i found, that i get best results with clear separation of what i want per category. keep the decades in decades and limit to a single result. only use a single grouping. etc. the plugin after all works with a whole mess of crowd-generated data.

about the regex:

it uses that regex on all the tags and when it matches it is considered a candidate for that category. it is a way to avoid hardcoding a list of "20s" 30s" etc strings. the default regex matches the two digit decades and iirc also four digit like "2000s".

limit:

the limit is applied to the endresult. it is absolute.

eg. when the plugin determines 3 tags via searchlist, gets another 2 from overflow, is prepended 1 more from another category, but the limit is 2... the result will be 2 tags: the one prepended and the first one from the overflow.

so when using prepend, make sure the limit is larger, than that of the category you are prepending. otherwise you just get the same values in both.

faq etc:

i marked a few issues with "faq", when i answer them here. i want people to be able to use the plugin, but on the other hand need to limit the time i need to spend on it. maybe i should open the github wiki section...
